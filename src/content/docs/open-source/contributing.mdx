Tolaria is open source. Contributions should preserve the project's core
constraints: local-first files, Git history, standards-based formats, and AI
readability.

## Before Changing the App

Read the architecture docs and relevant ADRs. Structural changes should align
with existing decisions or add a new ADR when the project requires one.

## Documentation Contributions

This docs site is a public-facing companion. Good docs contributions:

- Explain user workflows in clear language.
- Link back to upstream source material.
- Avoid leaking private vault content.
- Prefer curated examples over generated vault dumps.
- Keep route names stable.

## App Contributions

The app repository has its own development rules, test gates, localization
requirements, and CodeScene code-health gates. Follow the app repo instructions
when changing app source files.

## Proposal Workflow

This repository can be used to draft docs improvements first. Once a page is
stable, propose upstream changes to Tolaria's official docs or README where
appropriate.
